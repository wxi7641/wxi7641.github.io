Java笔记49

登录模块把 JDBC 小项目从“单纯数据库练习”推进到“具备基础访问控制的 Web 应用”。

登录注册为什么总会牵出过滤器

结合你导入的 Web 和 JDBC 笔记，这篇的关键知识链是：表单提交 -> Servlet 接收 -> Service 校验 -> Session 保存 -> Filter 拦截未登录请求。

用户模块适合拿来讲 Java Web 里的最基础登录流程；登录成功后，把用户对象放进 Session；过滤器统一拦截请求，判断用户是否已经登录；这样就不用每个 Servlet 都重复写权限校验代码。

JDBC 最值得整理的，不是单个 API 名字，而是一整条数据库访问链路：拿连接、写 SQL、设置参数、处理结果集、关闭资源。把这条线反复看熟，后面再学 MyBatis 就会明白框架到底帮我们省掉了哪些重复劳动。

表单提交、会话控制和拦截到底怎么配合

JDBC 相关内容一放进项目里，链路就会非常清晰：登录注册要查用户表，品牌管理要做增删改查，分页列表要查总数和当前页数据。也正因为这些需求很具体，JDBC 笔记特别适合围绕“一个功能怎么从页面走到数据库”来整理。

登录功能让我们看到查询单条数据时参数绑定和结果封装的写法；品牌列表和分页功能则把“查询多条数据”以及“前台展示”串了起来；这类案例最适合拿来解释 JDBC 为什么总和 Servlet、JSP 一起出现。

可以先看一段最能代表这一节的代码，很多概念放到具体写法里会更容易看懂。JDBC 代码最适合按“连接 -> SQL -> 参数 -> 结果 -> 关闭资源”这条顺序去读。只要顺序感建立起来，哪怕代码行数变多，也不会轻易看乱。

把这段代码拆开看，最关键的地方其实就在下面几处：

  • 登录成功后把用户对象放入 session，后续请求就可以通过会话识别当前登录人。
  • 过滤器统一拦截请求，避免每个业务 Servlet 都重复写登录校验代码。
  • 注册逻辑的本质是“先校验，再入库”，并把结果反馈给页面。

JDBC 这一层的细节看起来繁琐，但也正因为它足够原始，才特别适合理解数据库访问的本来面目。后面换成连接池、DBUtils 或 ORM 框架时，你会更清楚每一层抽象到底替你做了什么。

这一类功能最容易在哪些地方出错

这一块如果只是停在“会写”其实还不够，更有价值的是知道它还有哪些常见写法，以及它会继续和哪些知识连起来。

  • 在 Spring MVC 项目里，这类登录校验通常会转移到拦截器中完成。
  • 如果再往后学习前后端分离，就会从 Session 模式过渡到 JWT/Token 模式。
  • 这篇很适合和你导入的登录、JWT、Token 文档形成前后对照。

如果顺着这一篇继续往后补，下面这些方向都很值得继续展开：

  • 可以继续补连接池、DBUtils 和事务控制，让 JDBC 知识链更完整。
  • 涉及增删改查时，尽量把参数绑定和 SQL 组织方式分开解释，会更清楚。
  • 如果后面要衔接框架学习，这一类文章非常适合作为 MyBatis 的前置章节。
