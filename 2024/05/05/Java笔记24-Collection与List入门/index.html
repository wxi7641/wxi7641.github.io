Java笔记24-Collection与List入门

数组解决的是“长度固定的一组数据怎么保存”，集合解决的是“这组数据在运行过程中还会继续增删改查怎么办”。到了集合这里，Java 开始从基础语法真正走向更灵活的数据组织。

这一节最关键的变化只有一句话：集合把“长度固定”这件事放开了，但代价是必须更重视类型、遍历和结构选择。

🔬 从数据结构角度看，集合不是数组的语法替身

Collection 本身不是一个具体容器，它更像一套统一接口。真正把数据装进去的，是 ArrayList、LinkedList、HashSet 这些具体实现。这个设计的意义在于：上层代码只关心“能不能增删查遍历”，不必一开始就绑定某一种底层结构。

这一点和数组差别很大。数组是固定空间里的一段连续内存，类型和长度在创建时就已经确定；集合则把“存储规则”交给实现类，让结构能够在运行时继续演进。

对于 ArrayList 来说，变量里放的是对内部数组的引用，数组本体在堆中。随着元素增加，内部数组会经历扩容和拷贝，这也是“动态数组”这几个字真正的含义。

🔬 为什么 Iterator 不只是另一种遍历写法

Iterator 的价值不只是“少写几个字符”，而是它把遍历过程本身抽象成了一个统一协议。集合内部结构不同，但都可以用 hasNext() / next() 这一套接口来访问，这样上层代码就不用关心底层到底是数组还是链表。

这也是为什么 Iterator 在集合体系里特别重要。它不是单纯的遍历模板，而是集合最基础的访问抽象之一。很多集合在遍历期间如果结构被外部改动，还会触发 fail-fast 机制，防止遍历结果和内部状态彻底失配。

💡 容量增长和节点连接，背后其实是两种内存思路

ArrayList 依赖连续内存，所以读取快、定位快，但中间插入和扩容时要整体挪动或复制元素。LinkedList 依赖节点引用，所以插入删除更灵活，但每个节点都要额外保存前后指针，内存开销也更大。

从 CPU 视角看，数组的连续内存更容易命中缓存；链表则需要不断顺着指针找下一个节点，访问代价更高。这也是为什么“同样都是线性表”，它们在性能表现上会有明显差异。

fori 生成按索引遍历代码
iter 生成增强 for

🧠 小结

Collection 与 List 这一节的主线，是从数组的固定长度走向集合的动态结构。真正需要留下的不是方法名本身，而是三个判断：集合能不能动态增删、遍历时是否需要下标、底层结构会不会影响访问方式。
