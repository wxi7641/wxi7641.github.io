字符流更适合处理 文本内容。如果说字节流关注的是“底层怎么搬运字节”，那么字符流关注的就是“这些字节怎样按照字符来理解和输出”。

FileWriter 和 FileReader 的重点，是“文本”而不是“文件”

为什么字符流里总要配一个 char[]

Demo02 里这一段非常典型：

流类型 缓冲数组类型 更适合处理
字节流 byte[] 任意二进制数据
字符流 char[] 文本内容

JUnit 写文件读写测试，比直接塞在 main 里更清楚

Demo01、Demo02 用 @Test 包住方法，这本身也是一个很好的习惯：

这样做的好处是：

  • 每个功能点可以单独运行
  • 读和写可以拆成独立测试
  • 不必每次都从 main 方法组织一整段流程

真正继续往下走时，就会遇到编码问题

当前这批代码直接使用了 FileReader 和 FileWriter，这在默认编码和简单文本场景下完全够用。但一旦需要显式指定编码，后续更常见的做法会是：

  • InputStreamReader
  • OutputStreamWriter
