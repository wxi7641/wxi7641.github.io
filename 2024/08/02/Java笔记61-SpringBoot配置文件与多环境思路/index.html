Spring Boot 项目写到一定阶段，最先变复杂的往往不是业务代码，而是配置。端口、数据源、日志、静态资源、环境切换、上传大小限制，全都会往配置文件里汇聚。配置文件写得是否清楚，直接决定项目后期是不是容易维护。

配置分组 说明
spring.application.* 应用基础标识
server.* Web 端口、上下文等服务器参数
spring.datasource.* 数据库连接信息
mybatis.* MyBatis 的映射、别名、日志、驼峰转换

当配置项都进入一个文件之后，Spring Boot 工程会明显比多份 XML 更容易定位问题。

properties 和 yml 只是格式不同吗

两者都能完成 Spring Boot 配置，但写法风格不同。

格式 特点
.properties 直白、兼容性好、单行配置清晰
.yml / .yaml 层级表达更自然，复杂配置更好读

多环境配置不是“大项目专属”，很早就值得建立意识

即使是个人练手项目，也经常会遇到下面这些情况：

  • 本地开发库和线上库不一样
  • 开发端口与部署端口不一样
  • 日志等级在不同环境下应该不同

再配合：

  • application-dev.properties
  • application-test.properties
  • application-prod.properties

这样做的价值在于：同一套代码，可以在不同环境中加载不同配置。

配置项越来越多时，最怕的不是“麻烦”，而是失去边界

一个配置文件如果没有边界意识，常见问题会很快出现：

问题 表现
安全配置与普通配置混在一起 密钥容易泄露
本地路径写死 换机器就失效
环境差异不分离 开发、测试、生产互相污染
业务开关散落 排查行为变化非常困难

因此，配置文件真正要建立的不是“会写属性”，而是配置分层意识。
