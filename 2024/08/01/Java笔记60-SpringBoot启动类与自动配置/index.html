Java笔记60

Spring Boot 的启动类经常只有几行，但这几行代码几乎决定了整个应用怎样被拉起来。很多人第一次看到它时会觉得“太简单了”，真正往后写项目才会发现，简单只是表象，背后收起来的是大量默认配置、类路径判断和自动装配逻辑。

Spring Boot 真正改变的不是 Java Web 的分层结构，而是项目启动与基础整合的成本。

@SpringBootApplication 并不是一个“普通注解”

它本质上是组合注解，最常见的理解方式可以拆成下面三层：

组成能力 作用
@Configuration 表示当前类可以作为配置类
@EnableAutoConfiguration 根据依赖和配置项尝试自动装配
@ComponentScan 扫描当前包及其子包中的组件

也就是说，启动类放在哪个包下，不只是目录习惯的问题，而是会直接影响组件扫描范围。启动类通常放在根包，就是为了让控制器、服务层、Mapper、配置类都能被一并扫描到。

自动配置到底“自动”在哪

很多人刚接触时会把自动配置理解成“框架帮忙魔法生成代码”，其实并不是。更准确的说法是：

  1. 先根据依赖判断某个功能有没有出现
  2. 再根据配置项判断是否满足启用条件
  3. 满足条件后，注册一组默认 Bean

比如：

  • 导入了 spring-boot-starter-web，框架就会准备 Web 环境
  • 导入了 mybatis-spring-boot-starter，再配好数据源，MyBatis 相关能力就能接上
  • 配了 server.port，Tomcat 监听端口就会发生变化

这也是为什么 Spring Boot 非常强调“依赖 + 配置”的组合关系。

启动类很短，不代表可以“无脑复制”

启动类最常见的问题通常有这几种：

现象 常见原因
Controller 没扫描到 启动类包位置过低，扫描范围不对
Mapper 无法注入 忘记 @MapperScan，或包名写错
自动配置不生效 依赖不完整，或手工配置冲掉默认行为
应用启动报端口冲突 server.port 与本机占用端口冲突

因此，启动类虽然短，但不能把它当成“固定模板”。它依赖的是整个工程的包结构、依赖结构和配置结构。
