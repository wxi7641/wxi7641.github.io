Java笔记44

Spring MVC 解决的是表现层问题。它关注的不是“业务怎么做”，而是 HTTP 请求怎么进来、参数怎么绑定、结果怎么返回给页面或前端。

Spring MVC 的主线其实非常稳定：DispatcherServlet 接请求，Controller 处理请求，视图解析器或消息转换器决定结果怎么返回。

六、@ResponseBody 是页面型应用走向接口型应用的关键一步

控制器里还有一个很重要的点：

当 @ResponseBody 打开之后，返回值就不再交给 JSP，而是直接写入响应体。配合 mvc:annotation-driven，对象通常会被转换成 JSON。

这就是 Spring MVC 从传统页面开发走向前后端分离接口开发的关键接口位。

七、@RequestParam 的意义，在于把默认值和参数约束写进方法签名

分页参数写法很有代表性：

@RequestParam 这时解决的不只是取值问题，还包括：

  • 参数名映射
  • 默认值
  • 必填与否

相比手动判空，这种写法更接近“声明式参数处理”。

八、Spring MVC 最值得建立的是请求处理链思维

把这一套代码串起来，请求的大致流向其实很清楚：

如果是页面返回：

  • Controller 处理参数
  • 把数据放进 Model 或 ModelAndView
  • 视图解析器找到 JSP

如果是接口返回：

  • Controller 处理参数
  • 返回对象或数据
  • 消息转换器序列化成 JSON

九、这一节真正要掌握的不是注解名称，而是每个环节的职责

环节 负责什么
DispatcherServlet 统一接收请求
HandlerMapping 找到匹配的控制器方法
Controller 处理请求参数和业务入口
Model / ModelAndView 准备返回数据
视图解析器 定位页面
消息转换器 把对象转成响应内容

只要这条链真正理解了，后面的拦截器、异常处理、REST 风格、文件上传，其实都是继续往这条链上增加能力而已。
