Java笔记45

Spring Boot 真正让人觉得轻，不是因为它“变简单了”，而是因为很多以前必须手工拼起来的 Spring 配置，现在都被默认约定和自动配置接住了。

Spring Boot 不是弱化 Spring，而是把 Spring 项目最常见的初始化工作提前封装好。

四、自动配置到底替开发者省掉了什么

如果把传统 Spring + Spring MVC + MyBatis 项目和 Spring Boot 放在一起比较，最明显的差别通常有这些：

对比项 传统配置项目 Spring Boot
启动方式 容器 + XML 配置较多 main 方法直接启动
组件扫描 常常手工声明较多 自动扫描更集中
数据源配置 XML 或多处配置 统一进 application.properties
MyBatis 整合 需要更多显式配置 约定式配置更集中

自动配置的本质并不是“神秘地帮你做事”，而是框架根据依赖和配置项，提前把最常见的组合情况准备好了。

五、为什么 Starter 这么重要

Spring Boot 学习时经常会听到“起步依赖”这个词。它的真正意义不是一个新概念，而是把一类功能常用的依赖组合先打包好。

这样做有两个直接效果：

  1. 少写很多零散依赖
  2. 让自动配置有条件可判断

也就是说，Starter 负责把“需要哪些依赖”这件事先整理好，自动配置再负责决定“这些依赖该怎么配起来”。

七、配置文件除了集中，还开始承担“环境切换入口”的角色

Spring Boot 把配置收拢之后，后续最容易继续扩展的方向也就清晰了：

  • 多环境配置
  • 端口切换
  • 日志级别
  • 数据源切换
  • 文件上传限制

所以 application.properties 不只是“一个配置文件”，而是应用运行参数的主入口之一。

八、为什么 Spring Boot 让人有“从工程搭建回到业务开发”的感觉

以前做 Spring 项目时，很多精力会消耗在：

  • XML 配置
  • 组件注册
  • 数据源整合
  • Web 层基础装配

Spring Boot 出现之后，这部分重复劳动被压缩了很多，于是开发者就能更快把注意力放回：

  • 业务接口
  • 数据处理
  • 页面或接口输出

这也是为什么很多人第一次上手 Spring Boot，会直接感受到“项目终于没那么重了”。

九、Spring Boot 真正要建立的是“约定优于配置”的直觉

这一节最重要的不是记住多少属性，而是建立下面这层理解：

  • 依赖选得对，自动配置就能接上
  • 配置项放到规定位置，框架就知道去哪里找
  • 大多数默认行为已经足够覆盖常见项目
  • 需要个性化时，再去覆盖默认配置

只要这层直觉建立起来，后面继续往下学分页、文件上传、模板引擎、接口鉴权、多环境部署时，都会更顺。
