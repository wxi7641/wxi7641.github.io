Java笔记46

SSM 整合阶段真正要解决的，不是某个框架单独怎么写，而是 Spring、Spring MVC、MyBatis 三层能力怎样在同一个 Web 工程里稳定协作。到了这里，重点已经从单点语法变成整套架构的分工与配合。

SSM 的核心不在于配置文件数量，而在于容器、请求分发、持久层和事务终于被拼成了一条完整链路。

一、先把 SSM 里三层框架各自负责什么分清

最容易记的方式，仍然是职责划分：

框架 主要职责
Spring 管对象、管依赖、管事务
Spring MVC 接收请求、分发请求、返回页面或接口结果
MyBatis 执行 SQL、映射数据、管理持久层细节

SSM 整合的真正意义就在于：这三部分不再各跑各的，而是形成一条完整请求链。

Spring MVC 容器 Controller、视图解析、拦截器、静态资源等 Web 能力

这也是 SSM 整合阶段必须先建立起来的全局结构感。

方法类别 常见事务特征

新增 / 删除 / 修改 一般需要事务，传播行为更强

这说明事务在 SSM 里已经不再是“DAO 自己处理”的事，而是上升到了 Service 层的统一规则。

而数据源、事务、对象装配这些更外围的能力，已经被 Spring 接管了。

九、SSM 的主线应该始终用“请求穿三层”去理解

把所有配置拼起来后，一次典型请求的链路其实非常清楚：

这条链上每一层最好都能说清职责：

层级 负责什么
Controller 接请求、收参数、决定返回什么
Service 做业务组合、挂事务边界
Mapper 执行 SQL、映射数据

只要这条链不乱，SSM 项目就有非常强的可维护性基础。

十、为什么 SSM 既重要，又常被当作“过渡形态”

SSM 的价值非常大，因为它把传统 Java Web 分层思想真正落成了一整套工程实践：

  • Spring 负责容器和事务
  • Spring MVC 负责 Web 层
  • MyBatis 负责持久层

但它也会逐渐暴露一个特点：配置和整合成本还是偏重。于是后面才会顺势过渡到 Spring Boot。

所以学习 SSM 最好的心态不是把它当成“终点”，而是把它当成：

  1. 传统 Java Web 工程化的成熟形态
  2. 理解后续 Spring Boot 自动化整合的最好参照物

这也是为什么 SSM 总览更适合只讲框架结构和整合关系，而不必陷入具体业务模块细节。框架主线一旦看清，后面无论迁移、拆分还是升级，思路都会稳很多。
