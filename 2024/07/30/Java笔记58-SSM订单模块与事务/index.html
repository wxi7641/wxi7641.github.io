@Transactional 真正保护的不是某一行 SQL

很多人一开始会把事务理解成“数据库出错就回滚”。这个理解并不算错，但还不够完整。更准确的说法是：

事务保护的是一个完整业务动作的一致性，而不是孤立的一条语句。

所以事务为什么通常放在 Service 层，而不是 Controller 或 Mapper 层，也就很好理解了。

层级 是否适合作为事务边界
Controller 不适合，太靠近请求入口
Service 最适合，承接完整业务动作
Mapper 太细，只负责单次数据访问

为什么这类模块经常还会顺手带出多表查询

一旦进入“订单”“记录”“业务单据”这一类场景，展示层就很少只看一张表。实际页面经常需要同时看到：

  • 当前记录本身
  • 关联用户信息
  • 关联资源信息
  • 当前状态说明

这也正是为什么事务类模块经常会同时把“写操作”和“多表查询”一起带出来。

事务类模块和普通列表模块最大的差别在哪

模块类型 更关注什么
普通列表模块 展示、查询、分页
事务类模块 状态变化、一致性、业务边界

所以这类文章真正应该强调的，不是把每个业务步骤都贴出来，而是解释为什么这里需要事务、为什么这里的状态要由 Service 层统一控制。

分页和事务也经常会在同一模块里碰面

真实系统里，事务类模块往往既有提交动作，也有列表查询和历史记录展示。所以它很容易继续扩展到：

  • 当前用户记录列表
  • 管理端总览列表
  • 分页加载
  • 条件筛选

这说明事务模块并不是“只有写入”，而是读写两条线都会出现，只是写操作更强调边界和一致性。

更适合公开分享的写法是什么

像这种文章，公开整理时更适合保留：

  • @Transactional 的位置
  • Service 层为什么是业务边界
  • 状态更新为什么要统一收口
  • 多表展示为什么会出现

而不适合直接铺开：

  • 完整业务流程细节
  • 具体表结构隐私信息
  • 可直接照搬的核心业务实现

这样写出来，既能把事务讲明白，也能把模板项目真正保护住。
