Java笔记55

SSM 最容易让人混乱的地方，不是某个框架单独怎么写，而是三套配置怎样拼成一条完整运行链。把整合配置单独拎出来看，很多原本分散的知识点才会真正连起来。

SSM 的配置文件不是重复劳动，而是在描述容器分工、请求入口、数据接入和事务边界。

SSM 真正整合的是哪三层能力

最常见的理解方式仍然是按职责拆开看：

组件 负责什么
Spring 管对象、依赖注入、事务
Spring MVC 管请求分发、参数绑定、视图解析、拦截器
MyBatis 管 SQL、结果映射、持久层访问

单独学这三部分时，知识点是分开的；进入 SSM 之后，真正重要的是它们怎样合作。

容器 典型内容
Spring 根容器 dataSource、SqlSessionFactoryBean、Service、事务
Spring MVC 容器 Controller、视图解析器、静态资源、拦截器

很多初学者会把 applicationContext.xml 和 springmvc.xml 看成两份“差不多的配置文件”，其实它们承担的是两种不同层次的职责。

MapperScannerConfigurer 是 SSM 数据层整合的关键节点

它的作用不是“帮忙少写一行配置”，而是把 Mapper 接口代理对象自动注册进 Spring 容器。这样 Service 层才能自然注入持久层对象，而不需要手工创建。

把整套配置放在一起看，最重要的是建立结构感

可以把 SSM 的配置理解成下面这张图：

层次 入口 / 配置 最终作用
Web 入口 web.xml 拉起容器、接入请求
根容器 applicationContext.xml 管对象、数据源、事务、Mapper
MVC 容器 springmvc.xml 管 Controller、视图、拦截器
持久层核心 mybatis.xml 管映射规则、日志、别名

只要这张结构图清楚，后面再看登录、列表、分页、事务这些专题，就不会只看到一堆分散代码，而会知道每一段代码为什么放在那个位置上。
