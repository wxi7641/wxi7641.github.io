Java笔记43

Spring 最核心的价值，不是 XML 写法，也不是注解数量，而是它把 Java 开发中最麻烦的两件事统一处理了：对象该由谁创建，以及 公共逻辑该放在哪里。前者对应 IOC / DI，后者对应 AOP。

Spring 真正改变的，不是某个类的写法，而是整个应用组织对象和组织横切逻辑的方式。

一、IOC 先解决“对象到底该由谁来 new”

这三段代码放在一起，最重要的变化其实就是注释掉的那几行 new。Spring 的 IOC 想解决的正是这个问题：

  • Controller 不再自己创建 Service
  • Service 不再自己创建 DAO
  • 对象的创建和组装交给容器完成

这就是所谓的 控制反转。对象不是“自己去找依赖”，而是“由容器把依赖注入进来”。

这些逻辑并不属于核心业务本身，但又总想在方法前后执行。

六、动态代理把“增强逻辑”进一步通用了

JDK 动态代理的关键代码更像一个通用拦截器：

最值得理解的是 invoke()：

  1. 先执行增强逻辑
  2. 再调用真实方法
  3. 最后再执行收尾逻辑

这就是 AOP 最朴素的运行模型。

代理方式 特点
静态代理 代理类手写出来，结构直观但重复多
JDK 动态代理 面向接口生成代理，更灵活

Spring AOP 本质上也是在做类似的事情，只不过代理的创建和织入由框架统一接管了。

八、ApplicationContext 是 Spring 使用体验真正变化的入口

  • 对象已经提前由容器管理

  • 业务代码只需要拿来用

这就是为什么 Spring 学到后面，重点会越来越少放在“如何创建对象”，越来越多放在“如何声明对象关系”。

九、Spring 核心真正要建立的是两层直觉

第一层直觉是 IOC / DI：

  • 对象交给容器管
  • 依赖关系交给容器装

第二层直觉是 AOP：

  • 公共逻辑不要写进每个业务方法
  • 把横切逻辑抽离成统一增强

只要这两层真正想清楚，后面的事务管理、整合 MyBatis、整合 Spring MVC、再到 Spring Boot，都会变得非常顺。
