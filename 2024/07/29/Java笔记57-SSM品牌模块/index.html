Java笔记57

列表模块往往是 SSM 体系里最适合拿来观察分层结构的一类场景。因为它通常同时包含请求进入、服务层转发、Mapper 查询、页面渲染，甚至还会顺手带出一部分 AJAX 接口写法。

列表类模块为什么特别适合观察三层结构

一个典型的页面型控制器方法可能像这样：

虽然它表面上只是“查一个列表”，但其实把 MVC 结构的主线都带出来了：

层级 负责什么
Controller 接收请求、放数据、决定返回哪个页面
Service 承接业务调用
Mapper 执行查询
JSP / 视图层 展示最终数据

这比单独看配置文件更容易理解项目到底是怎么跑起来的。

Model 的作用不是“存一下数据”，而是承接视图渲染

很多初学者刚接触 Spring MVC 时，会把 Model 看成一个普通参数。实际上它在列表类页面里非常重要：

它完成的是从控制层到页面层的数据传递。控制器不需要自己拼 HTML，也不需要直接输出内容，而是把数据交给视图模板。

为什么列表模块经常顺手带出 CRUD 的接口化写法

列表一旦存在，后续最自然的需求通常就是：

  • 新增
  • 修改
  • 删除
  • 批量删除

这时控制层就很容易出现两种不同返回风格：

场景 更常见的返回方式
整页加载 返回 JSP / 视图名
弹窗编辑、局部刷新、批量操作 返回 JSON

这也是为什么列表模块特别适合用来理解“传统页面项目怎样逐步长出接口式能力”。

这一类模块最容易写乱的地方是什么

常见问题 更稳的做法
Controller 里直接写 SQL 查询留在 Mapper
页面逻辑和业务逻辑混在一起 页面留给 JSP，业务留给 Service
所有操作都整页跳转 局部交互可以逐步改为 JSON
只会查全部，不考虑条件和分页 列表模块最好继续扩展成真正的后台能力

列表模块看起来基础，但它非常适合用来观察一个 SSM 项目是否真正有清晰的边界感。

公开整理这类内容时，更值得强调的是“结构”，不是具体业务名

真正适合公开分享的不是某个品牌、某个表名或某个页面路径本身，而是这一类模块里稳定重复出现的模式：

  • 列表如何渲染
  • 数据如何跨层传递
  • 页面型请求和接口型请求如何并存
  • 基础 CRUD 如何逐步扩展

把这些模式讲清楚，比直接贴一整套业务页面源码更有长期价值。
