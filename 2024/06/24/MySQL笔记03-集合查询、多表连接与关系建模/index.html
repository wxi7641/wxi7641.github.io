MySQL笔记03

MySQL笔记03

单表查询解决的是“在一张表里找数据”，而真正接近业务场景的数据库操作，通常都会进入多表、关系和结果集组合的阶段。集合查询、多表连接、主外键建模，正是从“会查表”走向“会设计数据关系”的关键一步。

一、集合查询解决的是“把多段结果拼在一起”

这组 SQL 的开头先展示了三种很有代表性的集合操作：

最容易理解的方式是直接按集合意义去记：

操作 含义
UNION 并集，合并结果并去重
INTERSECT 交集，同时满足两边条件
EXCEPT 差集，保留左边减去右边后的结果

UNION 和 UNION ALL 不是一回事

虽然这组 SQL 里写的是 UNION，但继续扩展时一定会遇到 UNION ALL。

写法 特点
UNION 自动去重
UNION ALL 不去重，性能通常更直接

如果业务上允许重复记录保留，UNION ALL 会是更常见的选择。

集合查询也要注意环境兼容性

在一些 MySQL 环境中，INTERSECT 或 EXCEPT 可能无法直接使用。遇到这种情况，可以考虑改写成：

  • INNER JOIN
  • IN / NOT IN
  • EXISTS / NOT EXISTS

也就是说，集合查询是一种表达思路，不一定只能用单一写法实现。

三、笛卡尔积是多表查询里最应该先避开的情况

这段 SQL 的教学价值非常高，因为它会直接带出笛卡尔积问题。

如果两张表之间没有写连接条件，那么结果会是两张表记录的全排列。初学者一旦发现查询结果“突然爆炸变多”，十有八九就是这里出了问题。

多表查询一旦缺少连接条件，先怀疑是不是跑出了笛卡尔积。

四、内连接真正表示的是“两边都匹配上”

INNER JOIN 的核心语义很适合直接记成一句话：只保留两边能匹配上的记录。

在这个例子里，学生表和成绩表通过 id = sid 建立了关联，所以结果里看到的是“有成绩记录的学生”。

七、关系建模的关键不是表多，而是边界清楚

把部门和员工拆成两张表，真正体现的是关系型数据库最重要的一层思想：不同实体分表存储，再通过键建立联系。

这背后解决的是几个非常实际的问题：

  • 部门信息不需要在每个员工记录里重复写一遍
  • 部门名称修改时，不必批量改员工表
  • 查询时可以按关系重新拼接需要的数据

这也正是关系型数据库相比“把所有字段塞一张表里”更有结构感的地方。
