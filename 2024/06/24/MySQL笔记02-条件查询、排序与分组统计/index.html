MySQL笔记02

MySQL笔记02

真正让 SQL 变得有力量的，不是 SELECT 本身，而是“如何把要的数据筛出来”。条件查询、模糊匹配、排序、分页、聚合与分组，基本构成了数据库查询阶段最核心的能力。

一、先理解查询不是“把表全拿出来”

从这组 SQL 开始，重点已经不再是建表，而是怎样从已有数据里精准拿到想要的结果。

这一阶段最重要的意识是：查询并不一定返回整张表，而是根据当前问题挑字段、挑范围、挑顺序。

二、WHERE 是条件查询的真正入口

最常见的比较条件，在这一组 SQL 里都出现了：

比较条件最适合先按类型理解：

类型 代表写法 作用
比较运算 >、>=、=、<> 比较单个值
逻辑运算 AND、OR 组合多个条件
范围判断 BETWEEN ... AND ... 判断是否落在区间内
集合判断 IN (...) 判断是否属于某一组值
空值判断 IS NULL / IS NOT NULL 处理空值

AND 和 OR 的差别不是语法，而是结果范围

可以把它们理解成：

  • AND 是“同时满足”
  • OR 是“满足其一即可”

这也是为什么条件一复杂，最好加上括号，让逻辑边界更清晰。

IN 常常比连续多个 OR 更清爽

当条件本质上是“属于某一组值”时，IN 的可读性通常比多个 OR 更高。

三、LIKE 和 REGEXP 是两种不同层次的模糊匹配

这一组 SQL 很适合拿来比较它们的差异。

LIKE 更像简单通配

LIKE 的通配规则最常用的是：

写法 含义
% 任意长度字符
_ 单个字符

REGEXP 更像模式匹配

当条件从“模糊包含”升级到“满足某种字符规则”时，正则会明显更强。

LIKE 和 REGEXP 怎么选

场景 更适合的写法
前缀、后缀、包含查询 LIKE
指定模式、字符集合、长度规则 REGEXP

如果只是日常业务筛选，LIKE 往往已经够用；一旦要按更复杂的字符串规律匹配，REGEXP 的表达力会更强。

四、排序不是附属功能，而是结果组织方式

排序真正解决的问题是：查询结果以什么顺序展示。

多字段排序的关键在优先级

这句话的意思不是两个字段“同时等权排序”，而是：

  1. 先按 age 排
  2. 如果 age 相同，再按 math 排

这个优先级意识非常重要，因为很多报表和列表页都是这么组织数据的。

COLLATE 会影响文本排序规则

这说明文本排序并不只是“按字符大小比较”，而是受字符集和排序规则影响。中文、大小写、重音字符的排序行为，都可能跟这里有关。

五、LIMIT 把查询真正带进了分页思维

LIMIT 常见有两种理解方式：

写法 含义
LIMIT n 取前 n 行
LIMIT offset, size 从偏移量开始取 size 行

这也是大多数后台分页的起点。前端传页码和每页数量，后端最终都会落到类似的限制逻辑上。

六、聚合函数开始让查询从“取记录”变成“算结果”

这一组函数可以分成两类理解：

函数 作用
COUNT 计数
SUM 求和
AVG 求平均值
MAX 取最大值
MIN 取最小值

COUNT(*) 和 COUNT(col) 不能混着理解

写法 含义
COUNT(*) 统计行数
COUNT(address) 只统计该列不为 NULL 的行

这个差别在业务统计里非常关键。

七、GROUP BY 让“整体统计”变成“分组统计”

这里最值得理解的是：GROUP BY 不是简单排序，而是在“按某个字段把数据分桶”，每一桶再做统计。

多字段分组意味着分组键更细

当分组键从一个字段变成两个字段时，统计颗粒度会更细。

八、HAVING 和 WHERE 的差别一定要分清

最适合记忆的方式就是这一句：

WHERE 过滤行，HAVING 过滤组。

更直观一点可以看下面这张表：

关键字 作用阶段 是否常与聚合函数一起出现
WHERE 分组前 一般不直接对聚合结果过滤
HAVING 分组后 常用于过滤聚合后的结果

九、这一组查询语句最值得建立的是“执行顺序感”

很多 SQL 一旦写长，就会开始混乱。这个时候最有帮助的不是继续背新语法，而是建立查询执行顺序的直觉。

这条线一旦顺了，再回头看条件查询、分组统计、分页限制，就不会觉得它们是孤立知识点，而会知道它们分别在查询过程的哪个环节起作用。
