当并发代码开始从“会跑”走向“更可控”，synchronized 往往就不够用了。这个阶段最重要的两个方向，一个是 Lock + Condition，另一个是线程池。

并发工具类的价值，不是把写法变复杂，而是把等待、唤醒、复用、容量控制这些细节交给更合适的工具处理。

四、synchronized 和 Lock 怎么选，最好放在一张表里看

对比项 synchronized Lock / ReentrantLock
语法形式 Java 关键字 Java 接口及实现类
加锁释放 自动进入、自动释放 手动 lock() / unlock()

因此并不是 Lock 一定比 synchronized “高级”，而是它更适合复杂协作和更细粒度的控制。

五、线程池解决的，不只是“少写几个 new Thread()”

先看最常见的三种线程池工厂方法：

线程池的核心思想并不复杂：线程提前准备好，任务来了直接复用，而不是每来一个任务都重新创建线程。

这样做的意义主要有三点：

  1. 降低频繁创建和销毁线程的开销。
  2. 控制并发量，避免线程无限膨胀。
  3. 统一管理任务队列、线程数量和拒绝策略。

把这组参数读顺之后，线程池就不再只是“背定义”了，而是能真正看懂它怎么控流、怎么扩容、怎么拒绝新任务。

七、拒绝策略不是边角内容，而是线程池满载时的兜底方案

当前代码里把四种常见策略都注释出来了：

它们的差别最好直接对照着看：

拒绝策略 行为
AbortPolicy 直接抛异常
CallerRunsPolicy 交给调用者线程自己执行
DiscardPolicy 直接丢弃任务
DiscardOldestPolicy 丢弃队列里最老的任务，再尝试接收新任务

拒绝策略决定的是“线程池已经扛不住了之后怎么办”，所以它其实非常接近真实工程里的稳定性处理。

八、线程池参数读懂之后，容量关系也会更清楚

以这段配置为例：

可以顺着这个顺序理解任务进入线程池时会发生什么：

  1. 先用核心线程处理，直到 5 个核心线程都忙起来。
  2. 新任务继续进入阻塞队列，最多排 5 个。
  3. 队列满了之后，再扩容到最多 8 个线程。
  4. 还放不下时，才触发拒绝策略。

这就是线程池的工程意义所在：它把并发量从“无限放任”变成了“有边界地接收”。

十、从这里开始，并发编程已经进入“工具箱阶段”

前面的线程基础更像“先理解线程是什么”，这一节则开始回答：

  • 怎样更精确地协调线程
  • 怎样把等待队列拆开
  • 怎样让任务复用线程执行
  • 怎样限制并发规模

所以 Lock、Condition、线程池这三个知识点最好连起来看，它们共同指向的是一件事：让并发从能运行，走向可管理。
