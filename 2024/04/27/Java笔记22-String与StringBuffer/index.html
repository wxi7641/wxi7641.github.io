Java笔记22

字符串问题几乎是 Java 初学阶段最容易反复踩坑的部分，尤其是 ==、equals()、常量池、不可变对象和可变字符串这几条线，经常会混在一起。

先记住这句话：String 的值看起来能“变”，本质上每次变的往往是引用，而不是原对象本身。

String 为什么被称为不可变对象

真正理解 String，必须先理解“不可变”的含义。

例如：

concat("!") 不会直接把原来的 s1 改掉，而是返回一个新的字符串对象。如果没有重新接收返回值，原字符串内容并不会发生变化。

这就是 String 设计里非常重要的一点：

  • 字符串值稳定，适合共享和复用。
  • 常量池能减少重复对象创建。
  • 代价是频繁拼接时效率不占优。

也正因为这一点，可变字符串对象才会出现。

String、StringBuffer、StringBuilder 最好放在一张表里记

只记类名很容易混，放到一张对比表里会清楚很多：

类型 是否可变 线程安全 典型使用场景
String 否 值本身稳定 字符串常量、内容比较、普通文本处理
StringBuffer 是 是 多线程环境下的字符串拼接
StringBuilder 是 否 单线程环境下的高频拼接

通常日常开发里，StringBuilder 的使用频率会比 StringBuffer 更高；但从知识体系上看，StringBuffer 更适合帮助理解“可变字符串”这件事本身。

字符串这一节真正要带走的，不只是几个 API

字符串容易反复出问题，往往不是因为方法多，而是因为它同时牵涉：

  • 对象创建方式
  • 常量池
  • 引用比较
  • 内容比较
  • 不可变对象
  • 可变字符串

只要这几层关系理顺了，后面做表单校验、路径拼接、SQL 拼接、日志输出时，字符串处理就不会总是出意外。
